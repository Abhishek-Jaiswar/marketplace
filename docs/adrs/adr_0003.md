# ADR 0003: Database Indexing Strategy for Multi-Tenant Query Optimization

* **Status**: Accepted
* **Date**: 2026-07-08

---

## Context

In relational database systems like PostgreSQL:
1. Primary keys (`@id`) and unique columns (`@unique`) are automatically indexed.
2. Foreign keys (e.g. `userId`, `sellerId`, `mediaId`) are **not** automatically indexed by PostgreSQL.
3. Common query filters (e.g., `WHERE status = 'PENDING'`) and sorting columns require explicit indexes to avoid full table scans as the dataset grows.

In our multi-vendor marketplace, we have multiple cross-module relationships (e.g., `Seller` to `BankAccount` and `SellerDocument` to `Media`). Without explicit index definitions on foreign keys and filter columns, lookup queries and administrative dashboard lists will become slow, causing elevated database CPU usage.

## Decision

We will apply explicit single-column and composite indexes (`@@index`) in Prisma to all foreign keys and frequently filtered state fields:

### 1. Foreign Key Indexes (Avoid Full Table Scans on Joins)
* **`Store`**: Index `logoMediaId` and `bannerMediaId`.
* **`BankAccount`**: Index `sellerId`.
* **`SellerDocument`**: Index `verificationId` and `mediaId`.
* **`Media`**: Index `uploadedById`.

### 2. State & Status Indexes (Optimize Dashboard Filtering)
* **`Seller`**: Index `status` (for filtering active/suspended/draft sellers).
* **`BankAccount`**: Index `status` (for filtering pending bank verifications).
* **`SellerVerification`**: Index `status` (for filtering pending KYC reviews).
* **`SellerDocument`**: Index `status` (for checking individual document approvals).
* **`User`**: Index `status` (for checking user states).

### 3. TTL & Expiration Indexes (Optimize Cleanup Jobs)
* **`Session`**: Index `expiresAt`.
* **`VerificationToken`**: Index `expiresAt`.

---

## Scalability & Performance Analysis

### Pros
* **Fast Query Execution**: Prevents `Seq Scan` (Sequential Scans) on large tables when doing joins (e.g., resolving `Store.logo` or listing all `BankAccount` items for a seller).
* **High Concurrent Writes**: By indexing foreign keys, we prevent lock escalation during cascades (e.g., deleting a User cascades to their Sessions, which benefits from fast index lookups to delete related rows).
* **Optimized Admin Operations**: Admin queries filtering pending seller documents or verifications will execute in logarithmic time \(O(\log N)\) rather than linear time \(O(N)\).

### Cons
* **Write Overhead**: Every index adds a small write performance cost during `INSERT`, `UPDATE`, and `DELETE` operations because the database must keep the index tree updated.
* **Storage Footprint**: Indexes occupy additional disk space and database buffer pool (memory) capacity. This is a negligible trade-off for the performance gains.
